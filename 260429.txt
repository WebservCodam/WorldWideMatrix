‚óŹ Good, you're on changingServerPrinciples. Since you're driving the code,
  here's just the design sketch ‚ÄĒ responsibilities and the questions you'll hit,
   not implementation.

  Webserver (the orchestrator)
  - Owns: _epfd, the event loop, all Server instances, all Clients.
  - Maps: listenFd ‚Üí Server& (or Server*) for accepting, clientFd ‚Üí Server& for
  I/O.
  - Methods roughly: init() (build Servers from configs, open their sockets,
  register listenFds with epoll), run() (the epoll_wait loop, dispatches to
  connectNew/connectIn/connectOut), closeClient(fd), checkHealth().
  - Why it owns the clients map (not Server): when a client closes or times out,
   you remove it from epoll via _epfd and from the map. Keeping that single
  source of truth on Webserver avoids cross-class fd ownership puzzles.

  Server (one virtual server)
  - Owns: one ServerConfig, its own _listenFds (could be several listen
  directives in one config block).
  - Methods are config-driven, not fd-driven: things like "build a response for
  this request given my config," "does this URI match one of my locations," etc.
  - Importantly: Server does not call epoll_ctl or accept. It exposes behavior;
  Webserver drives the syscalls. That keeps the "Server only knows about config
  + request handling" invariant clean.

  Two judgment calls you'll run into

  1. Where does accept() live? Cleanest is Webserver: it has the epfd and the
  clients map, and accept just produces a new clientFd that needs to land in
  both the map and epoll. Server doesn't need to know. The Server reference gets
   attached at insert time (_clientFdToServer[clientFd] = 
  &_listenFdToServer[listenFd]).
  2. Where does the response get built? This is the one place Server earns its
  keep ‚ÄĒ Server::handleRequest(Client&) returns the bytes (or fills the client's
   write buffer). Webserver just handles the I/O around it. If you find Server
  has no methods beyond getters, that's a sign the split isn't pulling its
  weight and you've recreated option B with extra steps.

  Order I'd tackle it in
  1. Fix the Server server(_epfd); shadow in the current startServer first on
  the old branch (or carry the fix over) ‚ÄĒ otherwise you'll be refactoring
  around a latent bug.
  2. Rename Server ‚Üí Webserver, get it compiling and running unchanged.
  3. Introduce empty Server class, move _serverConfigs ownership into it (one
  Server per ServerConfig), keep all behavior on Webserver.
  4. Move listenFd creation into Server::init(), but Webserver still calls
  epoll_ctl to register them.
  5. Once stable, look for behavior that's actually config-dependent and migrate
   that into Server.

  Each step keeps the program runnable, which is the main thing when you're
  learning ‚ÄĒ you can stop at any point and still have a working server.

‚óŹ Good place to land. Before you dive in, a few framing questions and a
  suggested first slice ‚ÄĒ pick what fits, then go.

  Start small: GET a static file, 200 or 404, no extras. That alone exercises
  the full path (request ‚Üí match ‚Üí read file ‚Üí headers ‚Üí write) and gives you a
  working server. Add methods, autoindex, CGI, uploads, error pages, redirects
  on top once the skeleton holds. Trying to design for all of those up front
  tends to produce abstractions that don't fit the cases they were meant to
  handle.

  Where the response is built. This is the one slot where the Server split earns
   its keep. A reasonable shape:

  - Server::handleRequest(Client&) ‚ÄĒ takes the parsed HttpRequest from the
  client, walks its own ServerConfig to pick the matching LocationConfig, fills
  in client._response (status, headers, body). No I/O.
  - Client::serializeResponse() ‚ÄĒ already declared, currently returns nullptr.
  Job: turn _response into the wire-format std::string (status line + headers +
  CRLF + body). No config, no logic.
  - Webserv::connectOut ‚ÄĒ calls serializeResponse(), write()s, handles partial
  writes and the EPOLLIN flip. No knowledge of HTTP semantics.

  That keeps the layering clean: Webserv owns I/O, Server owns "what should the
  answer be," Client owns "what does the answer look like on the wire."

  Things to decide before writing code

  1. Where does location matching live? I'd put it on Server (since it owns the
  config) as something like findLocation(const std::string& uri). Pure function,
   easy to test.
  2. How do you signal errors? Two reasonable options: throw an exception with a
   status code, caught at the top of handleRequest and converted to an error
  response; or return early after setting _response.status. Pick one and stick
  to it ‚ÄĒ mixing both gets messy fast.
  3. Body in memory vs. streamed? Easiest first pass: read the whole file into
  _response.body. Fine for the static path. CGI and large uploads will push you
  toward a streamed model later, but don't pre-build for it.
  4. Partial writes. write() can return fewer bytes than asked. You'll need to
  track how much of the serialized response has been sent on the Client (e.g.
  _writeBuf + offset) and only flip back to EPOLLIN once it's drained. The
  current connectOut doesn't handle this ‚ÄĒ easy to forget.

  Suggested order

  1. Wire connectOut to call serializeResponse() instead of the hardcoded HTML,
  with a stub _response (status 200, hello-world body) ‚ÄĒ proves the I/O path
  still works.
  2. Implement serializeResponse() properly, covering status line + headers +
  body. No logic, just formatting.
  3. Add Server::handleRequest(Client&) that, for now, only handles GET on a
  literal file path from the URI relative to a hardcoded root. Sets _response,
  calls serializeResponse() indirectly via connectOut.
  4. Wire location matching into handleRequest ‚ÄĒ now root, index, and
  allowed_methods from the config start mattering.
  5. Add 4xx/5xx error responses and the error-pages directive.

  Each step keeps the server runnable. Where do you want to start ‚ÄĒ the wiring
  (step 1‚Äď2) or jump straight to a real handleRequest?
