# webserv

A web server is a program that waits for requests coming from clients (a web browser, or a tool like `curl`) and answers them by sending back web pages, images, or other files. Every time you open a website, a web server somewhere is what hands your browser the page you see.

**webserv** is our own version of one. The goal of the project is to learn how a real web server works by building it ourselves, from scratch, in C++.

## Description

When you start it, the program:

- reads a configuration file (a text file that tells it which addresses to listen on and how to answer requests),
- opens the network connections (sockets) described in that file,
- and then waits for requests and answers each one by:
  - parsing the request,
  - figuring out which website and which part of it the request is asking for,
  - and replying with the right response: a file saved on disk, a page generated by a script (CGI), the result of an upload, a redirect, or an error page.
  
webserv serves many clients at the same time on a **single thread**, using **non-blocking** I/O so it never gets stuck waiting on any one of them. Instead of helping one client at a time while everyone else waits in line, it uses a Linux feature called `epoll` to watch all the open connections together and only do work on the ones that are actually ready to be read from or written to.

Because the sockets are non-blocking, a read or write returns immediately with whatever data is available instead of pausing the program. So requests and responses are handled a piece at a time: webserv reads a bit more of a request whenever new data arrives, and sends a bit more of a response whenever the socket is ready to receive it. This way one slow client can never freeze the whole server. Connections can also be reused for several requests (keep-alive) when the client asks for it, and are closed automatically when they sit idle for too long.

The architecture splits into three parts:

- **Configuration parser** (`src/configparser/`): a lexer, parser and
  validator that turn an nginx-style config into validated `ServerConfig` objects, one per `server` block.
- **HTTP parser** (`src/httpparser/`): turns the raw bytes of a request into a
  structured `HttpRequest` (request line, headers, query string, body), with
  support for chunked transfer encoding.
- **Server core** (`src/`): `Webserv` runs the event loop; `Server` handles a request against its configuration; `Client` holds per-connection state; `Cgi` builds the environment for CGI scripts.

## Features

- HTTP/1.1 with **keep-alive** connections and per-connection idle timeouts
- Fully non-blocking I/O driven by a single `epoll` loop
- **Virtual hosts**: multiple `server` blocks can share one `host:port` and are told apart by the `Host` header (`server_name` in the configuration file)
- `GET`, `POST` and `DELETE` methods, configurable per location
- Static file serving with MIME-type detection
- **Directory listing** (autoindex) and configurable `index` files
- **File uploads** (`POST`) and deletion (`DELETE`) to a configurable path
- **CGI** support (e.g. Python scripts) via an `execve`-based interpreter call
- HTTP **redirects** via the `return` directive (3xx → `Location`)
- Custom and built-in **error pages**
- Per-server and per-location `client_max_body_size` limits
- Request **routing by location prefix**, with per-location `root` overrides

## Instructions

### Requirements

- A C++17 compiler (`c++` / `g++` or `clang++`)
- `make`
- Linux (the server uses the `epoll` API)

### Compilation

```sh
make        # builds the ./webserv binary
make clean  # removes object files
make fclean # removes object files and the binary
make re     # full rebuild
```

### Running

The server takes exactly one argument: the path to a configuration file.

```sh
./webserv config_files/valid/eval.conf
```

Then point a browser or `curl` at the address declared in the config. For the
sample `eval.conf` (which listens on `127.0.0.1:8085`):

```sh
# Fetch the landing page
curl http://127.0.0.1:8085/

# Browse a directory listing (autoindex)
curl http://127.0.0.1:8085/images/

# Upload a file, then delete it
curl -X POST --data-binary @somefile.txt http://127.0.0.1:8085/uploads/hello.txt
curl -X DELETE http://127.0.0.1:8085/uploads/hello.txt

# Follow a redirect
curl -iL http://127.0.0.1:8085/old
```

Ready-to-use configurations live in `config_files/valid/`:

- `eval.conf`: a single server exercising every feature, with matching pages
  under `www/eval/`
- `vhosts.conf`: two virtual hosts (`site_a`, `site_b`) sharing one address
- `example.conf`: a broader example including CGI

The `config_files/invalid/` directory holds configs that are intentionally malformed; the server rejects each of them with an explanatory error, which is
useful for testing the parser.

### Configuration format

The config follows nginx's syntax. A minimal example:

```nginx
server {
    listen 127.0.0.1:8085;
    server_name example;
    client_max_body_size 2m;
    root /www/;

    error_page 404 404.html;

    location / {
        root /www/eval/;
        index index.html;
        methods GET;
    }

    location /uploads {
        methods GET POST DELETE;
        upload_path www/uploads;
        autoindex on;
    }
}
```

Supported directives include `listen`, `server_name`, `root`, `index`,
`methods`, `autoindex`, `client_max_body_size`, `error_page`, `return`,
`upload_path`, `cgi_handler` and `keepalive_timeout`.
